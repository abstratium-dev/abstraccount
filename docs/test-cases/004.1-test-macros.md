# Test Case 004: Test Macros

**Feature:** Macro System - BankingExpense Macro  
**Date:** 2026-04-06  

## Preconditions

See [PRECONDITIONS.md](./PRECONDITIONS.md) for general preconditions.

**CRITICAL:** This test case depends on:
- [Test Case 001](./001-create-journal-with-accounts.md) - Journal and account tree must exist
- [Test Case 002](./002-open-the-books.md) - Opening balances must be established
- [Test Case 003](./003-record-initial-business-transactions.md) - Initial transactions recorded

## Test Objective

Verify that the macro system works correctly by testing them.

The test will validate:
- Macro selection and parameter form display
- Parameter validation and default values
- Transaction preview generation from macro template
- Transaction creation via macro execution
- Proper double-entry accounting for banking expenses

## Macro Under Test: BankingExpense

### Scenarios: Execute BankingExpense macro to record a bank charge

```gherkin
Feature: Macro System - BankingExpense

  Background:
    Given the user is signed into the application
    And the journal "Abstratium 2026" exists with a complete account tree
    And opening balances have been established for 2026-01-01
    And initial business transactions have been recorded
    And the user is on the journal detail page for "Abstratium 2026"

  Scenario: Record banking expense using BankingExpense macro
    When the user navigates to the "Transactions" section
    And the user clicks "Create Transaction from Macro" or "Use Macro"
    Then the macro selection interface should be displayed
    And the BankingExpense macro should be available in the macro list
    And the macro description should be displayed to help user understand its purpose
    
    When the user selects the "BankingExpense" macro from the dropdown
    Then the macro parameter form should be displayed
    And all 6 parameters should be displayed with appropriate UI controls
    And the form should show the following fields:
      | Field | Type | Default Value | Required |
      | Transaction date | Date picker | {today} | Yes |
      | Description | Text input | (empty) | Yes |
      | Invoice number | Text input | {next_invoice_PI} | Yes |
      | Amount | Number input | (empty) | Yes |
      | Bank account | Account dropdown | (empty) | Yes |
      | Financial expense account | Account dropdown | (empty) | Yes |
    And default values should be pre-filled where specified
    And required fields should be clearly marked
    
    When the user enters "2026-08-01" as the transaction date
    And the user enters "Test macros 004.1" as the description
    And the user accepts the default invoice number "{next_invoice_PI}"
    And the user enters "1.00" as the amount
    And the user selects account "1020 Bank Account (asset)" as the bank account
    And the user selects account "6900 Financial expense" as the expense account
    Then all required fields should be filled
    And the form should pass validation
    
    When the user clicks "Execute Macro"
    Then the transaction should be created successfully via the macro system
    And the transaction should be persisted to the database
    And the transaction should be balanced (debits = credits)
    And the journal should be displayed

    When the user views the newly created transaction
    Then the transaction should have the following properties:
      | Property | Value |
      | Date | 2026-08-01 |
      | Partner | P00000004 |
      | Description | Test macros 004.1 |
      | Status | Posted (*) |
      | Tags | Payment: |
    And the transaction should have 2 entries
    And entry 1 should debit account "6900 Financial expense" for CHF 1.00
    And entry 2 should credit account "1020 Bank Account (asset)" for CHF -1.00
    And the partner should be automatically set to P00000004
    And the tag `Payment:` should be automatically added
    And the invoice number should show the actual generated value
    And the transaction should have a unique ID
    And the transaction should be linked to the correct journal
    And all fields should be correctly stored
    
    When the user views account balances as of "2026-08-01"
    Then the "6900 Financial expense" account should show a current balance of 16.00
    And the "1020 Bank Account (asset)" account should a current balance of 1,984.00
    And the financial expense account (6900) should have increased by CHF 1.00
    And the bank account (1020) should have decreased by CHF 1.00
    And all other account balances should remain unchanged
    And the balance sheet equation should hold

  Scenario: Validate macro parameter filtering
    When the user selects the "BankingExpense" macro
    And the user clicks on the "Bank account" dropdown
    Then only accounts matching the pattern "1.*:10.*:100.*:1020.*" should be displayed
    And the bank account dropdown should only show accounts matching `1.*:10.*:100.*:1020.*`
    And account "1020 Bank Account (asset)" should be available
    And account "1000 Cash" should NOT be available
    And the dropdown should use filtered account selection
    
    When the user clicks on the "Financial expense account" dropdown
    Then only accounts matching the pattern "6.*:6900.*" should be displayed
    And the expense account dropdown should only show accounts matching `6.*:6900.*`
    And account "6900 Financial expense" should be available
    And account "6500 Administrative expenses" should NOT be available

  Scenario: Validate required field enforcement
    When the user selects the "BankingExpense" macro
    And the user leaves the "Description" field empty
    And the user attempts to preview or create the transaction
    Then a validation error should be displayed
    And the error should indicate "Description is required"
    And the error message should be specific and actionable
    And the transaction should not be created
    And required fields cannot be left empty
    
    When the user enters "Test macros 004.1" as the description
    But leaves the "Amount" field empty
    And the user attempts to preview or create the transaction
    Then a validation error should be displayed
    And the error should indicate "Amount is required"
    And the transaction should not be created
    And the amount must be a positive number
    And invalid inputs should trigger appropriate error messages

  Scenario: Validate default value substitution
    When the user selects the "BankingExpense" macro
    Then the "Transaction date" field should be pre-filled with today's date
    And the "Invoice number" field should be pre-filled with "{next_invoice_PI}" or the next available invoice number
    
    When the user completes all other required fields
    And the user creates the transaction without modifying the defaults
    Then the transaction should use the current date
    And the transaction should use the next sequential invoice number in the PI series

  Scenario: Verify transaction appears in transaction list
    Given the user has created a transaction using the BankingExpense macro
    When the user views the transaction list for "Abstratium 2026"
    And the user filters by date "2026-08-01"
    Then the transaction "Test macros 004.1" should appear in the list
    And the transaction should be filterable by date, partner, and tags
    
    When the user filters by partner "P00000004"
    Then the transaction "Test macros 004.1" should appear in the list
    
    When the user filters by tag "Payment:"
    Then the transaction "Test macros 004.1" should appear in the list
    And tags should be properly stored and queryable
```

## Acceptance Criteria

- [ ] User can access the macro selection interface
- [ ] User can select the macros
- [ ] Parameter form displays all required fields with correct types
- [ ] Default values are pre-filled (e.g. date, invoice number)
- [ ] Account dropdowns are filtered according to macro definition
- [ ] Required field validation works correctly
- [ ] Amount validation accepts positive numbers only
- [ ] Tags are automatically added
- [ ] User can create transactions from macros
- [ ] Created transactions are properly balanced
- [ ] Account balances are updated correctly
- [ ] Transactions appear in transaction list
- [ ] Balance sheet equation holds after transactions

## Notes

- These tests validate the **macro system infrastructure** 
- The macros simplify common tasks (e.g. recording bank fees) by:
  - Pre-setting the partner
  - Auto-adding the relevant tags
  - Providing filtered account selection
  - Generating invoice numbers automatically

### UI/UX Considerations

- The macro interface should be intuitive and easy to use
- Parameter form should provide helpful prompts and validation feedback
- Account dropdowns should show account codes and names
- Error messages should be specific and actionable
- Success feedback should confirm transaction creation
